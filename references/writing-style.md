# 专业报告说人话规则

本文件只约束用户最终看到的文字。判断依据与评分过程仍按各自 Reference 执行。

## 总原则

- 结论前置。先回答匹不匹配、能不能用、好不好用、值不值得装。
- 正面表达。知道什么就直接说什么，不预设用户误解。
- 一句只讲一件事。长内容拆成平行短句或列表。
- 同一事实只出现一次。身份、数据、场景和建议各放在自己的位置。
- 使用普通人可理解的词，可以保留专业技术术词，但不得是黑话；荋 必须保留的技术词，第一次出现时需要用一句朴实的人话清晰解释。
- 文风尽量幽默、轻松、可爱，但要平等、客观并基于事实，不能摆高姿态，不能高高在上教育人，不能拿费用、风险、隐私和用户知识水平开玩笑。

## 六条说人话规则

1. **先说事实，再下判断。** 写“需要安装 Python 和 6 个依赖包”，不写“具备一定的环境配置门槛”。
2. **主语必须清楚。** 判断与建议优先使用目标 Skill 的真实名称、`它`、`用户`或`你`。页首 Skill 定位第一句写出名称，第二、三句再次指代时用“它”，第四句允许重新写出名称作收束。其他卡片或段落第一句写出名称后，后续再次指代时用“它”。出现其他可能混淆的对象时，重新写出 Skill 名称。指标句可以直接用指标名称作主语。“适合场景 / 不适合场景”是短语列表，不套用主语规则。
3. **动作必须具体。** 写“用户要补三次资料”，不写“需要一定程度的人工参与”。
4. **能说结果，不说状态。** 写“项目文件没写清能否商用”，不写“使用许可未验证”。只有确实无法确认且无法进一步解释时才用“未验证”。
5. **一句只做一件事。** 连续出现两个以上逗号时，优先拆句。数字、条件和结论不要挤在一句话里。
6. **相信读者。** 同一事实只出现一次，不在段尾换一种说法再总结，不替读者制造误解后再纠正。

## 专业性从哪里来

- 用具体文件、数字、步骤、限制和来源支撑判断；
- 区分“官方说明”“根据规则判断”“安装实测”和“长期体验”；
- 不确定时指出缺什么证据，不用模糊话补洞；
- 轻松感来自清楚、自然和有节奏，不来自段子、夸张和网络口癖；
- 费用、权限、隐私、许可证和最终建议保持克制。

## 结论区固定写法

### Skill 定位

用 3—4 个短句：

1. `【Skill 名称】是【真实身份】。`
2. `【Skill 名称】通过【主要方式】完成【核心任务】。`
3. `最终用户可以得到【主要结果】。`
4. `经评估，【Skill 名称】更适合【典型用户】。`（适用对象不明显时再写）

页首 Skill 定位第一句必须以目标 Skill 名称开头。第二、三句如果再次指代目标 Skill，使用“它”；如果主语是用户或你，则直接写明。第四句可以重新写出目标 Skill 名称作收束。禁止用“这是一个”“这是一套”开头，禁止把身份、方式、任务、依赖和结果塞进一个长句。

### 章节摘要

“这个 Skill 是啥”和“这个 Skill 好用吗”各保留一行摘要，保证章节完整：

- “这个 Skill 是啥”概括真实身份、主要用途和核心执行能力；
- “这个 Skill 好用吗”概括整体完成度、主要优势和最大使用门槛；
- 摘要只报结论，五格、双栏和指标表再展开细节；
- 两行摘要可以与页首或结论概述重复核心判断，但不得复制完整句子或证据。

### 最终结论

用户提供需求时，第一句必须带目标 Skill 名称：

- `【Skill 名称】匹配你的需求。`
- `【Skill 名称】部分匹配你的需求。`
- `【Skill 名称】不匹配你的需求。`

用户未提供具体需求时，不使用“目标用户需求”这类内部话。按人群判断直接写：

- `【Skill 名称】适合这类用户。`
- `【Skill 名称】部分适合这类用户。`
- `【Skill 名称】不适合这类用户。`

推荐条旁必须写清“这类用户”是谁，例如：`判断对象：准备安装 Skill 的普通用户`。

后续只补 1—3 个短句：核心理由、最重要限制、是否值得安装。禁止绕着说。

### 量化分析

用 2—4 个平行短句，只摘取影响采用决定的数字，分别写：

- 公开关注度与维护情况；
- 安装复杂度、任务复杂度与额外涉及费用；
- 核心任务的 Token 消耗；
- 风险值。

指标句直接用指标名称作主语：写“安装复杂度得分为 4.8/10”“Token 消耗得分为 8.8/10”。不要用一个分号长句串完所有数字。

Skill 调用准确性是定性结果，放在指标表，不塞进量化分析卡。

### 适用场景

用 2—4 个平行短句说明：

- 最适合什么任务；
- 最适合什么人；
- 能交付什么结果。

这里不写安装步骤，不重复当前用户为什么不匹配。

### 使用建议

第一句必须写清对象与动作：

- 不推荐：`不建议安装它。`
- 谨慎采用：`建议先试用它，再决定是否安装。`
- 有条件推荐：`如果【具体条件】，可以安装它。`
- 推荐或强推荐：`建议安装它。`

禁止只写“有条件安装”“谨慎试用”这类标签句。条件必须说明需要什么 AI agent、软件、账号、材料或操作投入。

后续只说明什么时候可以考虑、先用什么材料或功能轻量试用、什么情况停用。

### 这个 Skill 适合你吗

顶部结论直接复用“最终结论”第一句与“使用建议”第一句。不得另写第三套结论。

- 不匹配：`【Skill 名称】不匹配你的需求。不建议安装它。`
- 部分匹配且需谨慎：`【Skill 名称】部分匹配你的需求。建议先试用它，再决定是否安装。`
- 部分匹配且有明确条件：`【Skill 名称】部分匹配你的需求。如果【具体条件】，可以安装它。`
- 匹配：`【Skill 名称】匹配你的需求。建议安装它。`

按目标人群判断时改为：

- `【Skill 名称】不适合这类用户。不建议安装它。`
- `【Skill 名称】部分适合这类用户。建议先试用它，再决定是否安装。`
- `【Skill 名称】部分适合这类用户。如果【具体条件】，可以安装它。`
- `【Skill 名称】适合这类用户。建议安装它。`

下方只列“适合场景”和“不适合场景”。每项直接写任务、偏好或使用条件，不写 Skill 名称、“它”“用户”，也不重复“适合”“不适合”。

### 五张功能卡

“使用时机、用户提供、任务步骤、额外需要、最后交付”全部使用短句列表：

- 每张卡写 2—5 条，每条只写一个对象或动作；
- 每条不超过 35 个汉字；
- 写“你要学习的书籍文件或本地文件夹”，不写“文件或路径”；
- 写“每课讲义与习题 / 知识摘要 / 学习路线状态”，不把多项交付挤进一句。

## 禁用表达

最终报告不得出现：

本文件是完整写作规则。渲染器只自动拦截其中部分高频表达，不能代替交付前人工质检。

- `不要把……理解成……`
- `并不是……而是……`
- `装上后，你最多……`
- `核心工作仍需其他能力完成`
- `可能会带来一定的……`
- `从某种程度上来说`
- `值得注意的是`
- `总体而言`
- `综上所述`
- `赋能`、`抓手`、`闭环`等没有新增信息的商业套话
- `具备一定的……能力`
- `需要一定程度的……`
- `在……方面表现良好`
- `可实现……`
- `核心路径通常不必`
- `联网必须，Key 可选`
- `按典型目标用户判断`
- `此分数不代表所有人`
- `接受静态、只读的采用评估`
- `目标用户需求`
- `采用代价`
- `证据边界`
- `这是一个`、`这是一套`
- 单独使用的`有条件安装。`
- 面向普通用户使用`仓库`，改写为“GitHub 项目”“项目文件”或“项目代码”
- `PDF 路径`、`Markdown 路径`、`核心路径`、`推荐路径`、`全开路径`

“路径”只用于真实的文件地址、网页地址或文件夹位置。表达不同用法时，直接写清条件和动作：写“学习 Markdown 文件时不用注册”，不写“Markdown 路径不用注册”；写“转换 PDF 时需要 MinerU”，不写“PDF 路径需要 MinerU”。

遇到不匹配，直接写“它不能完成什么”。遇到有条件推荐，直接列条件。

## 常见翻译

| 生硬表达 | 说人话 |
|---|---|
| 核心路径通常不必注册 | 通常不用注册账号 |
| 联网必须，Key 可选 | 必须联网；只有启用某项服务才需要 Key |
| 使用许可未验证 | 项目文件没写清能否商用 |
| 公开关注刚起步 | 目前只有 X Stars、Y 次提交，公开使用反馈不多 |
| 当前版本偏向 Claude Code | 说明按 Claude Code 写；换到其他 AI agent 需要改路径或命令 |
| 涉及的数据与隐私 | 哪些内容会发到外部服务 |
| 可能存在兼容性问题 | 尚未确认能否在 Windows 运行 |
| 典型目标用户 | 准备安装 Skill 的人；或直接写清具体人群 |
| 静态、只读评估 | 只看说明和公开资料，不安装、不运行 |
| 采用代价 | 要装什么、花多少钱、费多少 Token |
| 证据不足 | 目前没找到测试、价格或真实案例 |
| Markdown 路径可不使用 | 学习 Markdown 文件时不需要 |
| PDF 路径需要 MinerU | 转换 PDF 时需要 MinerU |
| 环境匹配 | 能不能在你的电脑和 AI agent 上用 |
| 交付匹配 | 最后给你的东西是不是你要的 |

## 技术词翻译

优先使用以下表达：

| 避免直接写 | 改写为 |
|---|---|
| `multi_agent`、多 agent、子 agent | 多个 AI agent 协作 |
| LLM | AI 模型 |
| CI | 自动测试流程 |
| PR | 代码合并申请 |
| repo | GitHub 项目、项目文件或项目代码 |
| TDD | 先写测试，再写功能 |
| hook | 自动触发步骤 |
| runtime | 运行环境 |
| Python 数据库 | Python 依赖包；若确为数据库，写具体产品名 |

`CLI`、`API`、`MCP`、`Key` 首次出现时，用一句话说明它在本 Skill 中负责什么。禁止只堆缩写。

GitHub 项目没有明确许可证时，只能写“代码公开，暂未发现直接费用。项目文件没写清能否商用。”不得把“公开可下载”改写成“核心免费”。

## 长度限制

- 结论区每条不超过 55 个汉字；确有数字或产品名时可略超。
- 其他列表每条只表达一个判断，优先控制在 80 个汉字内。
- 结论区不用分号。出现两个以上逗号时，优先拆句。
- 一个卡片写 2—4 条，不写整段。

## 交付前删除测试

逐条检查：

1. 删掉后是否不影响判断？不影响就删。
2. 同一事实是否在两个位置出现？只留最合适的一处。
3. 能否把“侧面影响”改成“直接结论”？能就改。
4. 普通人第一次看到技术词是否知道它做什么？不知道就翻译。
5. 是否为了轻松而牺牲准确？是就收回玩笑。
6. 是否出现抽象判断，却没有具体对象、动作或结果？有就改成人、软件或服务实际做了什么。
7. 是否把同一事实分别写进结论、指标、卡点和适合区？只保留职责最匹配的一处。

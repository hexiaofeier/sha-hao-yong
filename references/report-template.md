# 固定 HTML 报告模板

最终交付为单文件 HTML。结构和样式由 `scripts/render_report.py` 唯一控制。

## 强制生成方式

1. 运行 `python scripts/render_report.py init <Skill名称> --output-dir <交付目录>`，生成完整 JSON 骨架；
2. 只填写生成的过程 JSON，不另写字段和页面结构；
3. 运行 `python scripts/render_report.py build <过程JSON路径>`；
4. `build` 一次列出全部校验问题；按清单修改 JSON 后重新运行；
5. 校验通过后，脚本自动把 HTML 生成到 `过程文件_CO/` 的上一级目录，并完成输出检查；
6. 禁止直接手写 HTML、CSS，禁止渲染后人工改报告正文。

只检查 JSON、不生成 HTML 时运行：

```bash
python scripts/render_report.py check <过程JSON路径>
```

确需指定 HTML 路径时运行：

```bash
python scripts/render_report.py build <过程JSON路径> --output <最终HTML路径>
```

只有用户明确要求改版式时，才修改渲染脚本。普通评估任务不得重新设计页面。

## 固定页面结构

### 页首

- 报告标题；
- 评估日期与范围；
- Skill 定位：3—4 个短句，分别说明真实身份和主要用途、适用需求、工作方式、交付结果。要从用户的角度出发阐述。参考以下示例：
示例：
啥好用是一个对 Skill 进行安装前评估的工作流。
啥好用帮助用户判断目标 Skill 是否适合自己的需求。
啥好用通过只读检查候选材料，评估用途、安装要求、费用、Token 消耗和风险。
最终用户可以得到一份 HTML 格式的 Skill 评估说明书。



### 1. 结论概述

顺序固定：

1. 推荐程度、判断口径与 10 格色条；
2. **最终结论**：第一句直接写匹配、部分匹配或不匹配；再写是否值得安装、一个关键原因和一个决定性限制；
3. **量化分析**：公开关注度、安装复杂度、任务复杂度、额外涉及费用、核心任务 Token、风险值；
4. **适用场景**：最适合什么用户、什么任务、得到什么结果；
5. **使用建议**：是否建议安装、什么时候考虑、如何轻量试用。

四项结论使用固定 2×2 浅色卡片。每个卡片只放 2—4 个平行短句。

内容边界：最终结论只写是否匹配、是否值得安装、一个关键原因和一个决定性限制；量化分析只摘取影响决定的关键数字；适用场景只写典型人群、任务和交付；使用建议只写装不装、何时考虑和轻量试用方向。

### 2. 这个 Skill 是啥

固定一行结论：概括 Skill 的真实身份、主要用途和核心执行能力。

固定五格：

| 使用时机 | 用户提供 | 任务步骤 | 额外需要 | 最后交付 |
|---|---|---|---|---|
| 什么时候调用 | 你要提供什么 | 3—4 个实际步骤 | AI agent、本地软件或联网服务 | 交付结果及其可用方式、主要作用 |

- **使用时机**：用 2—4 个短句写用户遇到什么情况时应该调用；
- **用户提供**：用 2—4 个短句写链接、文件、材料和可选的个人条件；
- **任务步骤**：写 Skill 完成任务的 3—5 个核心动作，每条只写一个步骤。
- **额外需要**：用 2—4 个短句写 Skill 依赖的 AI agent、运行环境、本地软件或联网服务；不在这里展开安装清单。
- **交付结果**：用 2—4 个短句写 Skill 最终交付的结果、形式、可用方式和主要作用。

五张卡全部使用列表，每条不超过 35 个汉字。文件输入要写“文件”“本地文件夹”或具体链接，不写含义不明的“路径”。

### 3. 这个 Skill 好用吗

固定一行结论：概括 Skill 的整体完成度、主要优势和最大使用门槛；不判断是否匹配具体用户。

1. 定性评估
“好用之处 / 可能卡住”固定为左右两个卡片。

- **好用之处**：2—4 个平行短句，说明目标 Skill 的优势、亮点和好用之处；
- **可能卡住**：2—4 个平行短句，说明目标 Skill 可能产生的卡点、不足和风险。

2. 指标分析
固定为：
| 主要指标 | 评估结果 | 用户影响 |
|---|---|---|
| 公开关注度 | | |
| 下载量 | | |
| 安装复杂度 | | |
| 任务复杂度 | | |
| Skill 调用准确性 | | |
| 额外涉及费用 | | |
| Token 消耗 | | |
| 风险值 | | |

- **主要指标**：1）公开关注度写明可核验的 GitHub Stars、Forks、榜单或其他公开数据；2）下载量仅有公开数值时显示。
- **评估结果**：量化数据结果，如得分，简单/中等/复杂，低/中/高；评估方式等。
- **用户影响**：包括对评估结果的解读，以对于用户来说意味着什么的说明。

3. 安装准备
固定为：
| 前期准备 | 是否必须 | 用户影响 |
|---|---|---|
| 额外软件安装 | | |
| 额外注册/登录的账号 | | |
| API KEY/联网服务 | | |
| 涉及的数据与隐私 | | |


### 4. 这个 Skill 适合你吗

固定一行结论：由“最终结论”第一句与“使用建议”第一句自动组成，使用与第 2、3 部分章节摘要相同的左侧朱砂线摘要条；
示例：`示例 Skill 不匹配你的需求。不建议安装示例 Skill。`

“适合场景 / 不适合场景”固定为左右两个卡片。
- **适合场景**：2—4 个平行短语，直接写任务、偏好或使用条件；
- **不适合场景**：2—4 个平行短语，直接写不匹配的任务、偏好或限制。
两组都不写 Skill 名称、“它”“用户”，也不重复“适合”“不适合”。
如果用户提出了自己的需求，要围绕用户需求进行适合与不适合场景的分析。

### 5. 使用节奏建议

固定为 3—5 个阶段：第一次怎么试、什么时候扩展、何时复盘、什么情况停用。

### 页尾

顺序固定：

1. 折叠显示来源与尚不能确认的事项；
2. 浅色“说明”，使用同层级列表：第一条解释本报告中的 AI agent，后续各条说明采用边界和安全提醒；不单设“名词说明”。

版式固定使用“黑曜生漆”：页面底色 `#F6F1E7`，正文 `#101417`，主标题色 `#294C46`，辅助青 `#58A08C`，强调红 `#E64B3C`。禁止由过程 JSON 自定义配色。

- 页首右上使用青色圆斑，页尾左下使用朱砂圆斑；页首使用一条 6px 青色横线；页尾横线从中间断开，居中固定显示同色文字“啥好用skill•盒小Feier”；
- Skill 名和推荐分数使用强调红，标题后缀和推荐等级使用主标题色；
- 定位框、结论卡、功能卡、分析框和说明框使用青色左边条、浅白底、圆角和浅阴影；
- 章节摘要与匹配结论只使用朱砂红左边条，右侧不设边条；背景透明，无圆角、无阴影；文字统一加粗；
- 表格使用主标题色外框、12px 圆角和浅阴影；第一列使用主标题色，不加粗；两张表的表头文字固定为 `17px`；
- “来源与尚不能确认的事项”使用 `17px`，与信息卡标题一致；信息框与“使用节奏建议”保持 `34px` 间距；
- 推荐条使用固定 10 色，从 `#F4EDE8` 依次加深到 `#E64B3C`；只允许离散色块，不使用 CSS 渐变；
- 标题前横条、步骤序号、链接下划线和首尾横线统一使用辅助青。

正文、列表、表格内容和页尾说明使用 `15px`；页首定位摘要与“这个 Skill 适合你吗”摘要使用 `17px`。报告标题、章节标题、卡片小标题、表头和推荐分数保持各自字号，不随正文缩小。

## 字段口径

- 所有定量评分共用规则：见 [scoring-rubric.md](scoring-rubric.md)；
- 安装复杂度：见 [installation-score.md](installation-score.md)；
- 任务复杂度：见 [task-complexity.md](task-complexity.md)，必须按典型核心任务评分；
- Skill 调用准确性：见 [invocation-accuracy.md](invocation-accuracy.md)，根据名称、描述和触发规则作定性判断；
- Token 消耗：见 [token-score.md](token-score.md)，必须按典型核心任务评分；
- 风险值：见 [risk-score.md](risk-score.md)；
- 推荐程度：见 [fit-assessment.md](fit-assessment.md)；
- 所有用户可见文字：见 [writing-style.md](writing-style.md)。

许可证未验证时，额外费用不得写“核心免费”或“无新增费用”；应分别说明直接费用和使用许可。

推荐分数、最终结论第一句和使用建议第一句必须一致：

| 推荐分数 | 个性化判断第一句 | 目标人群判断第一句 | 使用建议第一句 |
|---|---|---|---|
| 0—3.9 | Skill名称不匹配你的需求。 | Skill名称不适合这类用户。 | 不建议安装它。 |
| 4.0—5.9 | Skill名称部分匹配你的需求。 | Skill名称部分适合这类用户。 | 建议先试用它，再决定是否安装。 |
| 6.0—7.4 | Skill名称部分匹配你的需求。 | Skill名称部分适合这类用户。 | 如果满足具体条件，可以安装它。 |
| 7.5—10 | Skill名称匹配你的需求。 | Skill名称适合这类用户。 | 建议安装它。 |

`basis_type` 只在过程 JSON 中使用：`personalized` 表示结合具体用户判断，`target` 表示按写明的人群判断。用户可见的 `basis` 必须直接说明依据：

- 个性化示例：`根据你的行研需求、Windows 和 Codex 环境判断`；
- 目标人群示例：`判断对象：准备安装 Skill 的普通用户`。

没有证据时写 `未验证`。下载量没有公开数值时，不创建该指标。

## 五个区域的唯一职责

| 区域 | 只负责 | 不负责 |
|---|---|---|
| 结论 | 是否匹配、是否安装、决定性理由 | 完整功能介绍、详细步骤、全部限制 |
| 这个 Skill 是啥 | 适用用户、调用场景、输入、工作过程、依赖与交付 | 推荐、评分和使用评价 |
| 这个 Skill 好用吗 | 整体完成度、质量表现、量化指标、卡点和安装准备 | 用户匹配与推荐结论 |
| 这个 Skill 适合你吗 | 当前需求与能力、交付、环境和代价的差异 | 重复指标、安装清单和完整功能 |
| 使用节奏建议 | 第一次试用、扩展、复盘和停用动作 | 再次解释为什么推荐 |

## 重复规则

### 允许重复

只允许四类摘要与详情重复：

1. 总结论与“这个 Skill 适合你吗”顶部结论必须完全一致；
2. 量化分析只报关键结果，指标表解释分数和原因；
3. 页首定位与“这个 Skill 是啥”一行结论可以概括同一身份，但五格只在第 2 部分展开；
4. “这个 Skill 好用吗”一行结论可以概括优势和门槛，完整依据只在双栏与指标表展开。

### 禁止重复

- 身份、工作方式和交付只在“这个 Skill 是啥”完整说明；
- 软件、账号、Key 和联网要求只在安装准备列清单；
- 分数原因只在指标表展开；
- 质量优势和真实卡点只放好用与卡点双栏；
- 许可证、测试、维护和兼容性等证据缺口只在“尚不能确认”列明；
- 第一次试用、扩展和停用动作只在使用节奏建议出现；
- 结论可以概括一个决定性限制，但不得复制其完整证据。

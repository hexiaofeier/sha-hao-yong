# 能力形态与工具/产品关系

本文件用于内部判断。面向用户时只输出自然语言名称，不显示层级代码。

## 五种能力形态

| 能力形态 | 主要组成 | 核心判断 |
|---|---|---|
| 方法指令型 | 指令与方法 | AI agent 主要按方法推理和生成 |
| 资料增强型 | 方法 + 规范、案例、模板或素材 | 任务需要时加载支撑资料 |
| 本地执行型 | 脚本、命令行工具、运行环境或本地软件 | 电脑需要实际运行程序 |
| 联网服务型 | API、MCP、远程服务或外部账号 | 核心结果依赖联网服务或外部数据 |
| 复合工作流型 | 多工具、多服务、多 Skill 或多个 AI agent 编排 | 核心链存在条件分支、状态、关卡、交接或多段交付 |

普通方法有很多步骤、章节或检查项，仍可能只是方法指令型。只有技术执行链真实发生多节点编排时，才判为复合工作流型。

模式差异明显时分别写最短核心路径、推荐路径和全开路径，不取最高形态概括全部用法。

## 真正能力在哪里

先问：**完成核心任务时，真正干活的能力主要在哪里？**

| 自然语言结论 | 判断方法 |
|---|---|
| 能力主要在 Skill 自身 | 方法、资料或自带脚本已构成核心能力，不绑定指定产品也能完成任务 |
| Skill 搭配通用工具 | Skill 教 AI agent 使用可独立存在、可以替换的 CLI、软件或通用服务 |
| Skill 是指定产品入口 | 不安装、注册或购买指定软件、插件或 SaaS，核心能力就不成立 |
| 当前对象是 Skill 合集或平台 | 对象组织、分发或运行多个可独立识别的 Skills，应拆到具体 Skill 再评 |

## 非 Skill 对象

- 对象是 App、CLI、库或 SaaS 时，先写真实身份，不把整个产品强写成某种 Skill；
- 仓库只有一个薄 Skill 入口时，单独评该入口与产品的关系；
- “仓库很大”“功能很多”“有插件系统”不能证明它是 Skill 合集；
- 一个仓库同时有产品和使用说明 Skill 时，分别评价。

## 功能范围

- 单一功能：一个明确输入，交付一种主要结果；
- 串联功能：围绕一个目标完成 2—3 个连续阶段；
- 复合功能：覆盖多个任务或多种交付物；
- 套件/平台：包含多个 Skill、AI agent、工具或完整产品。

## 推广倾向

只有证据表明 Skill 自身增量有限，主要作用是引导注册、订阅或采用指定平台时，才写“有推广倾向”。出现产品链接本身不是证据。

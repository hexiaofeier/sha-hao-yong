# 推荐程度：10 分制

推荐程度回答“这个 Skill 对当前用户或写明的适用用户值不值得采用”。0 表示不推荐，10 表示强推荐。共用计算和报告规则见 [scoring-rubric.md](scoring-rubric.md)。

## 两种推荐

- 用户选择“需要结合我的需求评估”并回答“你希望这个 Skill 能帮你做什么”：输出 **个性化推荐程度**；
- 用户选择“按一般目标用户进行评估”：根据官方定位界定一般目标用户，输出该人群的推荐程度，并在报告中直接写清这类用户是谁。

个性化推荐只使用用户实际写明的需求，判断目标和交付是否匹配。用户没有提供的设备、AI agent、预算、Token、隐私和操作偏好一律视为未知，不得自行补全，也不得把未知写成不匹配。

不得把适用用户写成必然满足所有条件的完美用户。环境、预算、隐私和使用频率仍需按常见真实情况判断。

## 五个维度

| 维度 | 最高分 | 只判断 |
|---|---:|---|
| 目标匹配 | 2.0 | 是否解决核心任务，而非只覆盖辅助环节 |
| 交付匹配 | 2.0 | 格式、质量、编辑性和验证方式是否满足要求 |
| 环境匹配 | 2.0 | 系统、AI agent、软件、账号和许可证是否支持 |
| 代价匹配 | 2.0 | 安装、任务操作、额外涉及费用、Token、风险、隐私和权限是否可接受 |
| 投入回报 | 2.0 | 使用频率和节省的时间是否值得安装与维护 |

每项写一句直接原因。

## 等级

| 分数 | 推荐结论 |
|---:|---|
| 0—3.9 | 不推荐 |
| 4.0—5.9 | 谨慎采用 |
| 6.0—7.4 | 有条件推荐 |
| 7.5—8.9 | 推荐 |
| 9.0—10 | 强推荐 |

## 硬条件

- 核心执行路径不成立、许可证明确不允许、必需环境缺失且不可替代：直接不推荐；
- 硬条件证据缺失：给推荐区间并标 `证据不足`；
- 有条件推荐必须在结论中列出具体条件，不能只写标签；
- 公开关注度和 Stars 是背景信息，不直接计入推荐分；下载量只有找到公开数值时才使用；
- 安装复杂度或任务复杂度低不等于值得装，风险低也不等于结果可靠。

## 结论写法

必须同时写：

1. 推荐程度与分数；
2. 本次判断对象或具体判断依据；
3. 适用场景；
4. 主要限制；
5. 一句明确的采用建议。
